Personal warehouse credentials allow you to connect to your data warehouse using your own individual credentials instead of shared project credentials. This provides better security and audit trails for data access.

## When are personal warehouse credentials used?

<Info>
    Personal warehouse credentials are only used for projects that require user
    credentials. If a project doesn't require user credentials, it will use the
    shared project connection instead.
</Info>

Your personal warehouse credentials will be used when:

-   A project is configured to require user credentials for data access
-   You're running queries, creating charts, or building dashboards in that project
-   You're using the SQL runner or exploring data in projects with user credential requirements

## Setting up personal warehouse credentials

To configure your personal warehouse credentials:

1. Navigate to your user settings by clicking on your profile in the top right corner
2. Go to the "My warehouse credentials" section
3. Enter your individual warehouse connection details
4. Save your credentials

## Supported authentication methods

Depending on your warehouse type, you can authenticate using different methods:

### Snowflake

- **Sign in with Snowflake (OAuth)**: Use your Snowflake account to authenticate via OAuth. This requires your organization to have configured a Snowflake security integration. See [Snowflake connection setup](/get-started/setup-lightdash/connect-project#sign-in-with-snowflake) for details.
- **Private Key**: Authenticate using a private key file.
- **Password**: Use your Snowflake username and password.

### Databricks

- **Sign in with Databricks (OAuth)**: Use your Databricks account to authenticate via OAuth. This requires your organization to have configured a Databricks OAuth application. See [Databricks connection setup](/get-started/setup-lightdash/connect-project#sign-in-with-databricks) for details.
- **Personal Access Token**: Use a Databricks personal access token.

### BigQuery

- **Sign in with Google (OAuth)**: Use your Google account to authenticate via OAuth.
- **Service Account**: Use a service account JSON key file.

### Redshift

Redshift supports two authentication types for personal warehouse credentials. Use the **Authentication type** selector to choose:

- **Username & password** (default): enter your Redshift username and password.
- **AWS IAM**: authenticate with temporary AWS credentials. Lightdash uses the IAM identity to mint short-lived Redshift credentials at query time, so no long-lived database password is stored.

When **AWS IAM** is selected, provide:

- **AWS access key ID** — the `AWS_ACCESS_KEY_ID` value from your terminal.
- **AWS secret access key** — the `AWS_SECRET_ACCESS_KEY` value from your terminal.
- **AWS session token** — the `AWS_SESSION_TOKEN` value. Required for AWS SSO and other temporary credentials.

The easiest way to obtain these values is to run:

```bash
aws configure export-credentials --format env-no-export
```

For a named SSO profile, add `--profile your-profile`.

#### Advanced IAM options

An **Advanced IAM options** section (collapsed by default) exposes:

- **Database user** — optional. When left blank, Lightdash uses `GetClusterCredentialsWithIAMCommand` to mint credentials directly from the IAM identity instead of requiring an explicit Redshift database user.
- **Assume role ARN** — optional. An IAM role for Lightdash to assume before requesting Redshift credentials, for example `arn:aws:iam::123456789012:role/lightdash-ecom-redshift`.
- **Assume role external ID** — optional. Only required when the IAM role's trust policy enforces an external ID.

IAM credentials must include either an assume-role ARN or an access key ID and secret access key together.

<Note>
    The IAM policy, role, and trust setup are the same as for project-level
    Redshift IAM connections. See [Setting up AWS IAM for
    Redshift](/get-started/setup-lightdash/connect-project#redshift) for the
    required policy statements and role configuration.
</Note>

All sensitive Redshift IAM fields (`accessKeyId`, `secretAccessKey`, `sessionToken`, `assumeRoleArn`, `assumeRoleExternalId`) are cleared from API responses in the same way passwords are, and must be re-entered when editing your credentials.

### Other warehouses

For other warehouse types (Postgres, Trino, ClickHouse), authentication is done using username and password credentials.
