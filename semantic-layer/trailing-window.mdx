A rolling average over the last N rows, turning spiky daily data into a readable trend line.

<Info>**Status:** 🟡 Partial — row-based, not calendar-day-based. Gaps in dates mean the window spans more than 7 calendar days.</Info>

## The question

> "What's the 7-day rolling average of daily active users this month?"

## Why it matters

Daily metrics are noisy. Tuesdays spike, weekends dip, a product launch creates a one-day surge. A rolling average strips away the noise and reveals the underlying trend — is engagement actually growing, or was it just a good Wednesday?

## Example

### Source data

`daily_activity` table — raw DAU counts:

| activity_date | daily_active_users |
|---------------|--------------------|
| 2024-03-01    | 1,240              |
| 2024-03-02    | 890                |
| 2024-03-03    | 920                |
| 2024-03-04    | 1,410              |
| 2024-03-05    | 1,380              |
| 2024-03-06    | 1,290              |
| 2024-03-07    | 1,150              |
| 2024-03-08    | 1,320              |
| 2024-03-09    | 940                |
| 2024-03-10    | 870                |

### YAML

```yaml
models:
  - name: daily_activity
    columns:
      - name: activity_date
        meta:
          dimension:
            type: date
      - name: user_id
        meta:
          metrics:
            daily_active_users:
              type: count_distinct
```

Then add a **table calculation** in the Explore view:

```sql
avg(${daily_activity.daily_active_users}) over (
  order by ${daily_activity.activity_date}
  rows 6 preceding
)
```

## How to set it up

1. Define a date dimension and a daily metric in your YAML model.
2. Open Explore, select the date (by day) and the metric.
3. Click **+ Table calculation** and enter the `avg(...) over (... rows 6 preceding)` window function.
4. Sort by date ascending.
5. Visualise as a line chart with both the raw metric and the rolling average as separate series.

## Workaround

The table calculation is **row-based**: if your data has no row for Saturday and Sunday, `rows 6 preceding` will cover 9 calendar days, not 7. For calendar-accurate windows, build a dbt model that fills gaps first:

```sql
with date_spine as (
  select date_day
  from {{ dbt_utils.date_spine(
    datepart="day",
    start_date="'2024-03-01'",
    end_date="'2024-03-31'"
  ) }}
),

filled as (
  select
    ds.date_day,
    coalesce(da.daily_active_users, 0) as daily_active_users
  from date_spine ds
  left join {{ ref('daily_activity') }} da
    on ds.date_day = da.activity_date
)

select
  date_day,
  daily_active_users,
  avg(daily_active_users) over (
    order by date_day
    rows 6 preceding
  ) as rolling_7d_avg
from filled
```

Define your Lightdash metrics on this gap-filled model instead.
